## 0. 시작 전 — 마스터플랜 검토 요약 (꼭 읽고 시작)

계획 자체는 규율이 좋다. 다만 만들기 전에 못 박아둘 4가지.

### ✅ 그대로 가져갈 강점
- **검증 우선**: 만드는 속도(AI가 당김) ≠ 검증 속도(절대 안 당겨짐).
- **비용 반영 백테스트 = 생명선 게이트**: 통과 못 하면 멈춘다.
- **변동성 중복 제거**: 점수(위험조정 모멘텀)에 1번, 비중(역변동성)에 1번. 끝. 점수에 또 넣지 않는다.
- **전략 보수성 = 가설(ON/OFF), 평가 보수성 = 최대**.

### ⚠️ 만들기 전에 못 박을 4가지 (검토에서 나온 보완점)

1. **생존편향(survivorship bias)이 가장 큰 위험.**
   현재 S&P500 종목으로 과거를 돌리면, "살아남아 지수에 든 회사"만 보게 되어 **결과가 실제보다 좋게 나온다.** 백테스트가 생명선인데 그 생명선이 낙관 편향이면 게이트가 약해진다.
   → **이번 버전 결정:** point-in-time(시점별) 구성종목 데이터는 비싸고 구하기 어려우니 **안 쓴다.** 대신 **(a)** 한계를 결과 리포트에 항상 박아 출력하고, **(b)** SPY 대비 초과수익도 생존편향으로 부풀려질 수 있음을 명시하고, **(c)** 백테스트 합격선은 보수적으로 높게 잡는다. 진짜 심판은 Phase 3 페이퍼(포워드) 테스트다. 백테스트 통과는 "강한 증거"가 아니라 **페이퍼로 넘어갈 최소 조건**일 뿐이다.

2. **"의미 있는 초과수익"을 숫자로 미리 정한다.**
   게이트를 "의미 있으면 통과"로 두면, 나온 결과에 맞춰 기준을 깎게 된다(자기기만). → **돌리기 전에** 합격선을 숫자로 적어두고(아래 Phase 2), 그 숫자로만 판정한다.

3. **표본이 적다는 걸 인정한다.**
   월 1회 리밸런싱 × 15년 ≈ 180개 표본. 통계적으로 강한 증거가 아니다. **게이트를 통과해도 "약한 증거"** 이고, 그래서 페이퍼 단계(Phase 3)로 한 번 더 거른다.

4. **OOS는 한 번 쓰면 소모된다.**
   아웃오브샘플 기간 성적을 본 뒤 전략을 다시 고치면, 그 기간은 더 이상 "안 본 기간"이 아니다. OOS 결과를 보고 손대는 순간 재검증용 새 기간이 필요하다.

5. **yfinance는 불안정하다.**
   레이트리밋·간헐적 이상치·API 변경이 잦다. 스케줄로 돌릴 거면 **캐싱 + 데이터 검증**을 처음부터 넣는다(Phase 1에 포함).

## 1. 기술 스택 & 폴더

- **언어/환경:** Python 3.11 (레포 `.python-version`/`uv` 기준), 의존성은 `uv` 또는 `pip`.
- **핵심 라이브러리:** `pandas`, `numpy`, `yfinance`, `lxml`(위키 표 파싱), `alpaca-py`(Phase 3), `pytest`(테스트).
- **위치:** 이 레포 안 `trading-system/` 하위. (모노레포의 기존 프로젝트들과 같은 패턴)

```
trading-system/
├── IMPLEMENTATION_PLAN.md   # ← 이 문서
├── config.py                # 운영자 결정 항목(손잡이) 한 곳에
├── data/
│   ├── __init__.py
│   ├── adapter.py           # yfinance 가격 + 위키 GICS 섹터 + 캐싱/검증
│   └── cache/               # 받은 데이터 로컬 저장(gitignore)
├── factors.py               # 팩터 계산(모멘텀, 변동성)
├── scorer.py                # 섹터중립 z-score 합산 + 역변동성 비중
├── backtest.py              # Phase 2: 게이트 ON/OFF, 비용, OOS  ★생명선★
├── executor.py              # Phase 3: Alpaca 페이퍼 주문

├── tests/                   # 각 단계 검증용
└── logs/                    # 실행 로그(gitignore)
```

> 원칙: **작게 만들고 매 단계 출력을 눈으로 확인.** 한 파일이 끝나기 전 다음 파일로 안 넘어간다.

### Phase 1 — 데이터 어댑터 (1~2일)
**파일:** `data/adapter.py`
**목표:** "가격·유니버스·섹터를 깨끗하게 가져오는" 한 곳을 만든다. 여기가 부실하면 위가 다 거짓이 된다.

- [ ] `get_universe()` — 위키 S&P500 표에서 티커 + **GICS Sector** 컬럼 가져오기(`pd.read_html`).
- [ ] `get_prices(tickers, start, end)` — yfinance **수정종가(adjusted close)** 일별. 배당·분할 반영된 값이라야 모멘텀이 정확.
- [ ] **캐싱:** 한 번 받은 가격은 `data/cache/`에 저장(parquet/csv) → 매번 재다운로드 안 함(레이트리밋 회피).
- [ ] **데이터 검증(조용한 실패 방지):**
  - 받아온 종목 수 / 요청 종목 수 비율 출력(많이 빠지면 경고)
  - NaN 비율, 가격 0 또는 음수, 비정상 급등락(예: 하루 ±50% 초과) 플래그
  - 섹터당 종목 수 출력 → **섹터당 최소 개수 가드**(너무 적은 섹터는 중립화에서 제외/경고)

**게이트:** 임의 날짜에 ① 유니버스 ~500개 티커+섹터, ② 가격 매트릭스(날짜×종목)가 결측 적게 채워짐, ③ 검증 리포트가 깨끗.
**검증:** `tests/test_adapter.py` — 캐시 히트/미스, 섹터 컬럼 존재, NaN 가드 동작 확인.

### Phase 1 — 팩터 & 스코어러 (2~3일)
**파일:** `factors.py`, `scorer.py`
**목표:** 마스터플랜 4장 그대로. **변동성은 점수에서 딱 한 번.**

`factors.py`:
- [ ] `momentum_12_1(prices)` — **최근 1개월 제외, 12개월 수익률**(룩어헤드/미세구조 회피용으로 -1).
- [ ] `volatility(prices, window)` — 일별 수익률 표준편차(연율화).
- [ ] `risk_adjusted_momentum = momentum_12_1 / volatility` — **점수용 핵심 신호.**
- [ ] (대안 비교용) 순수 `momentum_12_1` — Phase 2에서 어느 쪽이 나은지 비교. **둘을 쌓지 않는다.**

`scorer.py`:
- [ ] `sector_neutral_zscore(values, sectors)` — 각 팩터 z-score를 **섹터 안에서** 계산(섹터 평균 차감/표준편차 나눔). 핫섹터 몰빵 방지.
- [ ] `combined_score = Σ(wᵢ × zscore_sectorᵢ)` — 가중치는 `config.py`(기본 동일가중). 잘게 튜닝 금지.
- [ ] `inverse_vol_weights(selected)` — 1/변동성, 합 1 정규화. 이건 **점수가 아니라 리스크 관리** 목적(중복 아님).
- [ ] `rank_and_select(scores, top_n)` — 상위 N 선정 → 비중 부여 → CSV 저장.

**게이트:** 특정 날짜 랭킹·비중이 **납득 가능**(아는 종목이 말 되게 배치), CSV로 저장됨.
**검증:** `tests/test_scorer.py` — z-score 평균≈0, 비중 합=1, 섹터 중립이 실제로 섹터 평균을 빼는지.

### Phase 2 — 정직한 백테스트 ★★ 생명선 ★★ (가장 오래 걸림)
**파일:** `backtest.py`
**목표:** "비용 반영 + 안 본 기간에서도 SPY를 이기는가"를 정직하게 본다. **여기서 막히면 전체를 멈추고 재설계.**

**돌리기 전에 합격선을 숫자로 못 박는다(예시, 운영자가 확정):**
- 전체 기간 + 아웃오브샘플 **둘 다에서** 비용 차감 후 SPY 대비 **연 초과수익(CAGR) ≥ +3~5%p**
- **샤프 개선** 이 SPY 대비 양(+)
- **MDD**(최대낙폭)가 견딜 수준(예: ≤ -35%)
- → 셋 다 충족해야 통과. 하나라도 빠지면 멈춤.
- 단, 현재 S&P500 구성종목 백테스트는 생존편향으로 초과수익도 부풀려질 수 있다. 그래서 위 합격선은 "실거래 가능" 판정이 아니라 **페이퍼로 넘어갈 최소 허들**이다.

**구현 항목:**
- [ ] **월 1회 리밸런싱 시뮬:** 매월 말 점수 → 상위 N 선정 → 역변동성 비중 → 보유.
- [ ] **룩어헤드 차단(1순위):** 신호는 **리밸 당일까지** 데이터만, 체결가는 **다음 거래일 시가**.
- [ ] **비용 비관 가정:** 슬리피지 편당 5~10bp + **회전율** 반영(많이 갈아탈수록 비용↑).
- [ ] **생존편향 한계를 리포트에 항상 출력**(현재 구성종목 사용 명시, SPY 대비 초과수익도 가짜 알파로 부풀려질 수 있음 명시).
- [ ] **전략 보수성 = 가설로 비교(2~3개만):**
  - 추세 게이트(200일 MA 위만) **ON vs OFF**
  - 보유 종목 수 버전(예: 15 vs 20)
  - → **버전은 2~3개로 제한.** 파라미터 미세조정 금지(과최적화).
- [ ] **아웃오브샘플(OOS):** 데이터를 **앞부분(개발)/뒷부분(검증)** 으로 나눔. 앞에서 1등 버전 고르고, **뒤(안 본 기간)에서 재확인.** 1등은 반드시 OOS 재검증. OOS 결과를 본 뒤 전략을 수정하면 그 OOS는 소모된 것으로 보고, 새 검증 기간 없이는 다시 합격 판정하지 않는다.
- [ ] **지표 출력:** CAGR · 변동성 · 샤프 · MDD · **SPY 대비 초과수익** · 회전율 + 자산곡선 그래프.

**게이트:** 위 "합격선" 숫자 충족(전체 + OOS 둘 다). 통과 못 하면 → **멈추고 재설계.** 통과해도 생존편향 때문에 **페이퍼 테스트로 넘어갈 자격**만 얻는다. (못 이기면 인덱스(SPY)가 답일 수 있음.)
**검증:** 동일 입력에 동일 결과(재현성), 비용 0으로 두면 수익↑·비용↑하면 수익↓이 맞게 움직이는지(비용 로직 점검), 룩어헤드 일부러 넣어보면 성과가 비현실적으로 좋아지는지(차단 로직 점검).

### Phase 3 — 페이퍼 트레이딩 + 로깅 (포워드 시계 ON)
**파일:** `executor.py`, `logger.py`, `run_daily.py`
**전제:** **Phase 2 게이트 통과 후에만.** 백테스트로 정한 **그 버전 그대로** 돌린다(여기서 버전 바꾸지 않음).

- [ ] `executor.py` — 스코어러 결과 → Alpaca **페이퍼** 주문(목표 비중 맞춰 매수/매도). 부분체결·실패 처리.
- [ ] `logger.py` — 리밸런싱마다 **점수·체결·잔고** 기록(CSV/DB). 나중에 백테스트와 대조용.
- [ ] `run_daily.py` — 진입점. "오늘 리밸 날인가?" 판단 → 스코어 → 주문 → 로깅. **출력을 눈으로 확인 가능하게.**
- [ ] VPS/cron 스케줄. (미국장 = 한국 밤~새벽, 봇이라 무관)

**게이트:** 수 개월(여러 국면) 페이퍼 성과가 백테스트와 **크게 안 어긋남.**
**검증:** 로그에 매 리밸 점수·체결가·잔고가 남고, 월별 성과를 백테스트 기대치와 나란히 비교한 표가 나옴.

### Phase 4 — 소액 실거래 (검증 완료 후만)
**전제:** Phase 3까지 통과 + **모니터링/개입 가능한 환경**(마스터플랜상 전역 후).

- [ ] **킬스위치 먼저:** 자동 정지 조건(일/누적 손실 한도 초과 시 전량 청산·주문 중단)을 코드로 **거래보다 먼저** 넣는다.
- [ ] 잃어도 되는 소액 + 손실 한도 정의(`config.py`).

**게이트:** 킬스위치 동작 검증됨 + 손실 한도 정의됨 + 개입 가능 환경.

|---|---|---|

## 4. 조용한 실패 체크리스트 (매 단계 점검)

- **과최적화:** 버전 2~3개로 제한 + OOS 재확인 + 미세조정 금지.
- **룩어헤드:** 미래 정보 누수 — 백테스트 1순위 점검.
- **생존편향:** 현재 구성종목만 사용 → 결과 과대평가. SPY 대비 초과수익도 부풀려질 수 있음. 리포트에 항상 명시.
- **OOS 소모:** OOS 결과 보고 전략 수정 금지. 수정하면 새 OOS 필요.
- **비용 누락:** 수수료·슬리피지·회전율 미반영 시 수익은 환상.
- **숨은 쏠림:** 같은 위험에 여러 번 베팅 금지(변동성 중복 제거 유지).
- **데이터 부실:** yfinance 결측/이상치 — 어댑터에서 검증·캐싱.
- **킬스위치:** 실거래 전 자동 정지 조건 코드화.
